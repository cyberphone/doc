Saturn 3
Payment Demo
Payment Session Debug Information 

The following page shows the messages (here slightly edited for brevity) exchanged between a Merchant (Payee), Merchant Bank, Wallet (Payer), and User Bank (Payment provider).  For traditional card payments there is also an Acquirer (aka "card processor") involved. The numbers shown in the different steps are supposed to match those of the [SATURN] presentation.

Saturn uses a JSON based message notation described in [YASMIN].

Current mode: Hybrid Payment

1

The user performs "Checkout" (after optionally selecting payment method), causing the Merchant server returning a Wallet invocation Web-page featuring a call to the "navigator.nativeConnect()" [CONNECT] browser API. Then the invoking Web-page waits for a ready signal from the Wallet.

2

When the ready signal has been received the Merchant Web-page sends a list of accepted account types (aka payment instruments) and associated signed [SIGNATURE] "paymentRequest" objects to the Wallet:

This mode should preferably be indicated in the Wallet user interface. See [UIDEMO].

After an optional selection of account (card) in the Wallet UI, the user authorizes the payment request (typically using a PIN):

2b

The result of this process is not supposed be directly available to the Merchant since it contains potentially sensitive user data.  For an example turn to Unencrypted User Authorization.

4c

Now the User Bank (equipped with the "ProviderAuthority" and "PayeeAuthority" objects), can check the validity of the "AuthorizationRequest" including:

  • Verifying that the "recepientUrl" in the "AuthorizationRequest" matches the "serviceUrl" of the User Bank.
  • Verifying that the "publicKey" in the "signature" object of the "PayeeAuthority" object and the public key of the first (=signature) certificate in the "certificatePath" of the "ProviderAuthority" object are identical.
  • Verifying that the Merchant is vouched for by a provider belonging to one for the User Bank known trust network through the "certificatePath" in the "ProviderAuthority" object.
  • Verifying that the "publicKey" and "algorithm" in the "AuthorizationRequest" matches one of the "signatureParameters" objects in the "PayeeAuthority" object.
  • Verifying that the "publicKey" and "algorithm" in the "paymentRequest" matches one of the "signatureParameters" objects in the "PayeeAuthority" object.
  • Verifying that decrypting "encryptedAuthorization" returns a valid user authorization object including "signature".
  • Verifying that the "requestHash" in the user authorization object matches the hash of the "paymentRequest" object.
  • Verifying that the "paymentMethod" in the "AuthorizationRequest" and in the user authorization object are identical.
  • Verifying that the "timeStamp" in the user authorization object is within limits like -AllowedClientClockSkew to (AllowedClientClockSkew + AuthorizationMaxAge) with respect to current time.
  • Verifying that the "publicKey" and "accountId" match a User Bank customer account.
4d

If the user authorization object also holds RBA (Risk Based Authentication) data, this is where such data should be validated.

Note that the inclusion of RBA data means that a related previous "AuthorizationRequest" did not result in an "AuthorizationResponse" (indicating success), but in a "ProviderUserResponse" holding specific RBA request data. See Provider User Response for more information.

4e

After validating the "AuthorizationRequest" and checking that the User actually has funds matching the request, the User Bank reserves the specified amount including a reference to the "payee" and "referenceId" of the "paymentRequest".

4f

After a successful preceding step, the User Bank embeds the "AuthorizationRequest" in a newly created "AuthorizationResponse"object.

4g

Then a number of properties are added including "encryptedAccountData" which holds an encrypted version of the User account which can be used for possible refunds. Encryption is performed using the "encryptionParameters" of the payment "ProviderAuthority" associated with the Merchant. The following shows typical account data before encryption:

{
    "@context": "https://sepa.payments.org/saturn/v3#ad",
    "payerIban": "FR1420041010050500013M02606"
}
Note that encrypted account data always points to a real account which may differ from the card like ("virtual") account used by the customer.
